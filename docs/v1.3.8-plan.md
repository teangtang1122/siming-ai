## 任务 B：Plan Agent 接入建档

**负责人**：后端 Agent/建档 1 人

### 当前状态

- 三个占位工具 `extract_facts`、`resolve_targets`、`apply_candidates` 定义在 `backend/app/services/agent/planner.py:204-231`
- 在 `orchestrator.py:21` 的 `_NOT_IMPLEMENTED_TOOLS` 集合中被跳过
- 从未被 `detect_intent()` 自动触发（`build_plan_from_intent()` 对 `project_init` 返回 `None`）
- 真正的 cataloging 逻辑在 `backend/app/services/cataloging/orchestrator.py`（492 行，完整可用）

### 实施方案

**不注册为 workspace tools**，而是让 Plan Orchestrator 直接调用 cataloging service。

#### B1. 新增 cataloging plan step executor

**编辑**：`backend/app/services/agent/orchestrator.py`
- 从 `_NOT_IMPLEMENTED_TOOLS` 移除三个工具名
- 新增 `_execute_cataloging_step(step, plan, session)` 方法
- 当 `step.tool` 为 `extract_facts` 时：调用 `cataloging.orchestrator.run_fact_extraction(chapter_id, session)`
- 当 `step.tool` 为 `resolve_targets` 时：调用 `cataloging.orchestrator.run_candidate_resolution(chapter_id, session)`
- 当 `step.tool` 为 `apply_candidates` 时：调用 `cataloging.applier.apply_candidates_for_run(run_id, session)`

#### B2. 让 detect_intent 识别建档意图

**编辑**：`backend/app/services/agent/planner.py`
- 在 `detect_intent()` 中添加 `"project_init"` 到意图映射（当前返回 `None`）
- 修改 `build_plan_from_intent()` 对 `project_init` 调用 `plan_cataloging_init()`
- 更新 `_INIT_KEYWORDS` 匹配词：`建档`, `给项目建档`, `整理资料`, `提取事实`

#### B3. 改进 plan_cataloging_init 生成逻辑

**编辑**：`backend/app/services/agent/planner.py` 的 `plan_cataloging_init()`
- 当前 step args 只传 `chapter_ids`，需改为按章节生成独立 step 组
- 每章 3 步：`extract_facts:{chapter_id}` → `resolve_targets:{chapter_id}` → `apply_candidates:{chapter_id}`
- 章节间串行（后一章的 extract 依赖前一章的 apply）
- 无 chapter_ids 时自动查询项目所有章节

#### B4. 前端 AgentPlanView 适配

**编辑**：`frontend/src/components/AgentPlanView.tsx`
- 当前已支持 6 种 step 状态和详情展示，无需大改
- 确保 cataloging step 的 label 正常中文显示（已在 planner.py 中定义）
- 可选：为 cataloging 步骤添加进度百分比显示（基于已完成章节数/总章节数）

#### B5. SSE 事件桥接

**编辑**：`backend/app/services/agent/bridge.py`
- `detect_and_stream_plan()` 中 `execute_plan()` 的 SSE 事件已通用化
- 需确保 cataloging 步骤执行时的中间事件（fact_extracted, candidate_created）能透传到前端
- 方案：在 orchestrator 的 `_execute_cataloging_step` 中 yield 中间事件，bridge 层转发

### C1. 数据库模型

**编辑**：`backend/app/database/models.py`

新增 `ScheduledTask` 模型：

```python
class ScheduledTask(Base):
    __tablename__ = "scheduled_tasks"
    id = Column(String(36), primary_key=True, default=generate_uuid)
    project_id = Column(String(36), ForeignKey("projects.id"), nullable=False)
    name = Column(String(200), nullable=False)
    prompt = Column(Text, nullable=False)           # 要执行的提示词
    cron_expr = Column(String(100))                 # cron 表达式（可选）
    interval_minutes = Column(Integer)              # 间隔分钟（可选）
    tool_policy = Column(JSON, default=list)        # 允许的工具列表
    status = Column(String(20), default="active")   # active|paused|error
    last_run_at = Column(DateTime)
    last_run_status = Column(String(20))
    last_run_output = Column(Text)
    next_run_at = Column(DateTime)
    created_at = Column(DateTime, server_default=func.now())
    updated_at = Column(DateTime, onupdate=func.now())
```

## 任务 D：RAG 与上下文质量

**负责人**：后端 RAG 1 人

- 字符预算制（`ContextBudget`，`context_packer.py:31`），无 tokenizer

- 无"未读到关键资料"的 warning

### D1. 近似 token 预算

**编辑**：`backend/app/services/rag/context_packer.py`
- 新增 `estimate_tokens(text: str) -> int`：中文约 1.5 字/token，英文按空格分词

- `pack_context` 中用 token 估算替代字符计数
- 默认 token 预算：`max_total = 6000` tokens（约 8000 字符）

### D2. Pinned Context（不可被挤掉的上下文）

**编辑**：`backend/app/services/rag/context_packer.py`
- 新增 `PinnedContext` 配置：当前大纲节点、已选角色状态、最近章节摘要（最近 2 章）
- 在 `pack_context` 中，pinned 内容优先分配预算，不受 RAG 召回排序影响
- pinned 内容用 `score=999` 标记（已有 `pinned_chunks` 机制可复用）

**编辑**：`backend/app/services/workspace/tools/chapter_writer.py`

### D3. 召回评估样例

**新建文件**：`backend/tests/test_rag_recall_quality.py`

- 断言：给定 query 应召回特定 chunk（基于 content hash）

### D4. "未读到关键资料" Warning

**编辑**：`backend/app/services/rag/context_packer.py`
- 在 `PackedContext.warnings` 中新增检查：

## 实施优先级与依赖

```
阶段 1（可立即开始，无依赖）：
  A - 乱码修复（3 行编辑 + 1 个测试文件）
  F - 前端拆分（纯前端重构，不影响后端）

阶段 2（依赖阶段 1 的 A 完成）：
  D - RAG 改进（改动 context_packer + chapter_writer）
  E - 技能成熟化（改动 skills service + 前端）

阶段 3（独立，工作量最大）：
  C - 定时任务（新增模型 + router + engine + 前端页面）
  B - Plan Agent 建档（改动 orchestrator + planner + bridge）

阶段 4（依赖以上全部）：
  G - 发布收尾（版本号 + 打包 + 发布）
```

## 风险点

1. **任务 B 的 SSE 桥接**：cataloging orchestrator 有自己的 SSE 事件流，需与 agent plan 的 SSE 事件融合，可能需要适配层
2. **任务 C 的 cron 解析**：Windows 环境下 croniter 的行为需验证
3. **任务 F 的拆分**：大组件拆分容易引入状态管理 bug，需充分测试
4. **任务 D 的 token 估算**：无 tokenizer 的近似估算在边界情况下可能不准
