# 墨枢路线图

本文档用于把墨枢从“功能快速推进”管理到“稳定可发布产品”。优先级按用户能否稳定完成长篇小说工作流排序，而不是按功能新鲜程度排序。

## 当前定位

墨枢是一个本地运行的长篇小说 AI 工作台，核心价值是把章节、大纲、角色状态、关系、世界观、伏笔、写作偏好、模型与外部 Agent 工作流放进同一个项目，让 AI 在长篇创作中持续读到正确资料，并把结果写回权威数据库。

## 管理原则

1. 稳定优先于扩功能：建档、写作、保存、恢复失败必须先可靠。
2. 单机体验优先：普通用户不应被要求理解 Python、Node、MCP 或模型路由细节。
3. 数据安全优先：用户作品、API Key、本地数据库和模型目录不能被误删、误传或误打包。
4. 可诊断优先：长任务失败时必须能看到模型、provider、步骤、错误、可重试入口。
5. 文档跟版本同步：每次发布必须同步 README、版本号、发布说明和验证记录。

## P0：稳定核心闭环

目标：用户可以创建/导入作品，建档，写作，保存，失败后继续。

- [ ] 为 `LLMGateway` 模型选择优先级补单元测试。
- [ ] 为建档候选解析、归一化、去重、应用补测试。
- [ ] 为章节保存后文件镜像、索引刷新、角色/世界观更新补回归测试。
- [ ] 明确长任务状态机：running、waiting_confirmation、paused_on_failure、cancelled、completed。
- [ ] 给外部 Agent 写回流程补一条端到端冒烟验证。

验收标准：后端测试能覆盖模型路由、建档候选、章节写入三个高风险区域；失败任务可以从失败步骤继续，不需要整轮重来。

## P1：发布与安装体验

目标：普通用户下载 `Moshu.exe` 后能顺利启动、配置模型、创建作品。

- [ ] README 顶部更新到当前版本，并补齐 2.5.5 / 2.5.6 变更说明。
- [ ] 增加发布检查清单：版本号、前端构建、后端测试、打包、sha256、Release 附件。
- [ ] 处理生产/打包环境默认 `secret_key`，避免固定默认值进入可联网场景。
- [ ] 梳理 Windows 杀毒误报说明，区分未签名、PyInstaller、本地服务、模型进程。
- [ ] 验证旧品牌 `NovelWritingAgent.exe` 与新版 `Moshu.exe` 的更新兼容。

验收标准：新用户不读源码也能完成下载、启动、配置模型、创建作品；发布包内容和 README 描述一致。

## P2：产品可用性

目标：让用户能理解 AI 正在做什么，并能修正错误。

- [ ] 建档页面显示每章当前阶段、模型来源、失败原因、候选数量。
- [ ] 候选项支持按角色、世界观、关系、时间线筛选和批量处理。
- [ ] 章节写作前显示本轮选中的上下文摘要。
- [ ] 增加“本章是否冲突”检查入口，输出冲突来源与建议改法。
- [ ] 增加项目级记忆管理页面：新增、停用、检索、来源追踪。

验收标准：用户能看懂 AI 读取了哪些资料、为什么失败、下一步该点哪里。

## P3：本地模型与外部 Agent 增强

目标：降低模型配置门槛，并让本地 CLI/Agent 工作流保持一致。

- [ ] 本地模型推荐根据 CPU、内存、显存给出保守默认值。
- [ ] 本地模型下载失败支持断点续传、校验失败重试、镜像源切换。
- [ ] 外部 Agent MCP 自动配置失败时给出可执行修复建议。
- [ ] 为 Claude Code、Codex、OpenCode、Qwen Code 等 CLI 维护兼容性表。
- [ ] 统一内部 LLM 与外部 Agent 的任务日志格式。

验收标准：用户知道该选哪个模型；外部 Agent 失败时能定位是配置、权限、模型还是任务输入问题。

## P4：协作与社区

目标：项目能接收反馈，而不是只靠个人记忆维护。

- [ ] 启用 issue 模板：Bug、功能建议、模型适配问题。
- [ ] 建立 `good first issue` 列表，适合新贡献者处理文档、测试、小修复。
- [ ] 补贡献指南，说明运行、测试、打包、PR 描述要求。
- [ ] 为 Release 建立固定格式：重点变化、修复、兼容性、验证、已知问题。

验收标准：任何用户提交问题时都能带上复现信息；任何发布都能追踪验证记录。
